Desktop-приложение (Windows/Linux) для управления устройствами платформы
«Умный дом» Яндекса — аналог мобильного приложения «Дом с Алисой», но на
рабочем столе. Работает через официальный REST API `api.iot.yandex.net/v1.0`,
без реверс-инжиниринга и скрытых API.

**Зачем.** У «Дома с Алисой» нет desktop-версии — только телефон. Если вы
целый день сидите за компом и хотите переключить свет/сценарий без того,
чтобы каждый раз тянуться к телефону — официального решения для этого не
существует. Поэтому пришлось сделать аналог самому.

## Скачать (для тех, кто просто хочет пользоваться)

1. Откройте [Releases](https://github.com/BazZziliuS/ya-pult/releases/latest)
   и скачайте последнюю версию: `.exe` для Windows (просто установщик,
   ставится как обычная программа) или `.AppImage` для Linux (сделайте файл
   исполняемым — `chmod +x` — и запустите).
2. При первом запуске нажмите «Войти через Яндекс» и войдите под тем же
   аккаунтом, что и в мобильном приложении «Дом с Алисой» — устройства,
   комнаты и сценарии подтянутся автоматически.
3. Дальше приложение живёт в системном трее — крестик на окне не закрывает
   его, а сворачивает; полностью выйти можно через меню трея (ПКМ по иконке).

## Возможности

- **Вход через Яндекс ID** — тот же аккаунт, что и в «Доме с Алисой», без
  дополнительных сервисов и без встроенного API-ключа (регистрируете своё
  OAuth-приложение, см. ниже).
- **Боковое меню с категориями** — комнаты, группы, датчики, сценарии; выбранная
  категория заполняет основную область, а не одна длинная лента вперемешку
  (по мотивам [Desktop-Yandex.Home-App](https://github.com/onegamerstory/Desktop-Yandex.Home-App)).
- **Плитки вместо развёрнутых карточек** — клик по устройству/группе открывает
  модалку со всеми настройками, а не расписывает десяток слайдеров сразу на
  весь экран; группы устройств управляются одним кликом (например, сразу все
  лампы «Лестница»), а не по одной.
- **Управление под тип умения**: тумблер вкл/выкл, слайдеры яркости/громкости/
  влажности с живым перетаскиванием (без «прилипания» к старому значению),
  температура света с кнопками быстрого возврата к тёплому/холодному, выбор
  произвольного цвета через Hue-слайдер + пресеты + сохранённые сцены
  освещения, режимы через выпадающий список.
- **Показания датчиков** — температура, влажность, батарея и т.д. в отдельной
  категории сайдбара, независимо от того, в какой они комнате.
- **Быстрый поиск** (`Ctrl+K`/`⌘K`) по всем устройствам и сценариям — выбор
  устройства сразу переключает сайдбар на его комнату и открывает модалку
  настроек, выбор сценария — запускает его.
- **Системный трей** — окно сворачивается в трей вместо закрытия; в меню трея
  (ПКМ по иконке) — список последних включений/выключений устройств и групп
  с быстрым тумблером прямо оттуда, без открытия окна.
- **Несколько домов (households)** — если в аккаунте Яндекса больше одного
  дома, сверху сайдбара появляется переключатель; комнаты, устройства и
  группы фильтруются по выбранному дому (для одного дома переключатель
  скрыт, ничего лишнего в интерфейсе).
- **Проверка обновлений** — при старте приложение сверяет свою версию с
  последним [GitHub Release](https://github.com/BazZziliuS/ya-pult/releases/latest);
  если вышла новая — тост со ссылкой «Открыть». Ненавязчиво: любая ошибка
  сети/лимита GitHub API просто молча игнорируется.
- **Профиль и настройки** — в шапке виден логин подключённого аккаунта; из
  выпадающего меню — выход и настройки (язык интерфейса RU/EN, системные
  уведомления ОС, частота опроса устройств).
- **Уведомления и индикация состояния** — нативные уведомления ОС о потере
  сессии/связи и завершении сценариев, индикатор «обновлено N сек назад»,
  устойчивый порядок элементов в карточках (не «прыгают» между опросами).
- **RU/EN-интерфейс** — переключается в настройках, сохраняется между
  запусками.

## Стек

- **Electron** + **TypeScript** во всех процессах (main, preload, renderer)
- **electron-vite** — сборка и hot reload для разработки
- **React** + **HeroUI** (`@heroui/react`, на Tailwind CSS) — интерфейс
- **electron-builder** — сборка `.exe` (NSIS) / `AppImage`

    /src

## Регистрация OAuth-приложения на oauth.yandex.ru

Приложение не содержит встроенного API-ключа — вы регистрируете **своё**
OAuth-приложение и вставляете его `client_id` в `.env`. Это отдельная
сущность от токена устройства/навыка и не требует модерации Яндекса для
личного использования.

1. Откройте https://oauth.yandex.ru/client/new
2. Название — любое, например «ЯПульт».
3. Платформы: включите **Веб-сервисы**.
4. Redirect URI — тут два варианта в зависимости от того, что вам позволяет
   указать форма Яндекса:
   - **Поле редактируется** — впишите `http://localhost:4567/callback` (порт
     можно изменить, тогда поменяйте его и в `.env`; значение должно совпадать
     один в один, включая путь).
   - **Поле заблокировано** (у части аккаунтов оно нередактируемо и всегда
     равно `https://oauth.yandex.ru/verification_code`) — ничего страшного,
     просто оставьте значение по умолчанию в `.env.example`/`.env`
     (`MAIN_VITE_YANDEX_REDIRECT_URI=https://oauth.yandex.ru/verification_code`).
     Это официальная страница Яндекса «для приложений без своего редиректа» —
     она выводит токен пользователю текстом для ручного копирования, но само
     приложение перехватывает токен из адреса раньше, чем страница успевает
     отрисоваться, так что копировать ничего не придётся.
5. Раздел «Доступ к данным»: добавьте права
   - `Просмотр состояния устройств умного дома` (`iot:view`)
   - `Управление устройствами умного дома` (`iot:control`)
6. Сохраните приложение и скопируйте **Client ID** (Client Secret не нужен —
   используется implicit flow, без секрета; можно даже перевыпустить его,
   если случайно куда-то показали).
7. Скопируйте `.env.example` в `.env` и подставьте свои значения:
   ```
   MAIN_VITE_YANDEX_CLIENT_ID=<ваш client_id>
   MAIN_VITE_YANDEX_REDIRECT_URI=<то же значение, что в настройках приложения>
   ```

Приложение открывает модальное окно авторизации Яндекса, вы входите под своим
Яндекс ID (тем же, что в приложении «Дом с Алисой»), Яндекс перенаправляет
браузер на `redirect_uri` с токеном в `#access_token=...`. Если это
`localhost` — физически его никто не слушает, навигация **обязана** упасть
ошибкой соединения, это ожидаемо. Если это `oauth.yandex.ru/verification_code`
— страница загрузится по-настоящему, но окно закроется раньше, чем вы успеете
увидеть код на экране. В обоих случаях токен перехватывается в `auth.ts` через
события навигации окна (`will-redirect`/`did-navigate`/`did-finish-load` и т.д.
— см. комментарии в файле), а не через реально работающий сервер.

## Установка и запуск

```bash
npm install
cp .env.example .env   # и подставьте свои значения
npm run dev             # разработка с hot reload
```

Другие команды:

```bash
npm run build            # production-сборка в /out
npm run start             # предпросмотр production-сборки
npm run typecheck          # проверка типов main/renderer отдельно
npm run test                 # юнит-тесты (Vitest), разово
npm run test:watch            # юнит-тесты в watch-режиме
npm run dist:win                # .exe через electron-builder (NSIS)
npm run dist:linux                # AppImage через electron-builder
```

Тесты (Vitest + Testing Library) покрывают чистую логику без запуска
Electron: сравнение версий и парсинг ответа GitHub API (`updateCheck.ts`),
валидность токена (`tokenStore.ts`), кодирование/декодирование ошибок через
IPC (`shared/types.ts`), сортировку capabilities/датчиков, конвертацию
HSV↔hex, единицы измерения и склонения (`renderer/src/lib`), а также хук
`useOptimisticValue` — именно ту логику, из-за которой раньше был баг с
«зависанием» интерфейса при смене цвета лампочки.

## CI: сборка и релиз по тегу

`.github/workflows/release.yml` собирает `.exe` (Windows) и `.AppImage`
(Linux) и публикует их в GitHub Release при пуше тега вида `v1.2.3`
(или вручную — вкладка Actions → Release → Run workflow).

Для сборки нужны те же переменные, что и в `.env` — заведите их в
**Settings → Secrets and variables → Actions → Repository secrets** под
именами `MAIN_VITE_YANDEX_CLIENT_ID` и `MAIN_VITE_YANDEX_REDIRECT_URI`
(значения — как в разделе про регистрацию OAuth-приложения выше).

## Особенности реализации и ограничения API

- **Таймаут навыка** — Яндекс ждёт ответ от устройства/облака производителя не
  дольше 3 секунд на своей стороне; если устройство «зависло», Яндекс сам
  вернёт ошибку по конкретному действию — карточка устройства покажет её,
  не блокируя остальной интерфейс.
- **Длина ответа/токена** — токен ограничен 2048 символами (проверяется в
  `tokenStore.ts` перед сохранением), тело ответа API — 5000 символами;
  наш HTTP-клиент это не режет, но `yandexApi.ts` рассчитан на такие объёмы.
- **Хранение токена** — только в зашифрованном виде через `safeStorage`
  Electron, в `app.getPath('userData')`. Если `safeStorage` недоступен в ОС
  (нет системного хранилища ключей), приложение откажется сохранить токен и
  запросит логин заново при следующем запуске.
- **401 от API** — токен сбрасывается автоматически (`main.ts:withAuthGuard`),
  UI откатывается на экран логина.
- **Refresh-токен** — implicit flow Яндекса его не выдаёт, токен просто
  протухает по истечении `expires_in`. Поле `refreshToken` в `StoredToken`
  зарезервировано: если понадобится долгоживущая сессия, замените
  `openAuthWindow` в `auth.ts` на authorization-code flow
  (`response_type=code`) с обменом кода на токен через свой backend/локальный
  сервер (client_secret нельзя держать в desktop-клиенте без бэкенда).
- **Групповые действия** — реальный путь `POST /groups/{id}/actions`, а не
  `/devices/groups/{id}/actions`, как указано в официальной документации
  (второй вариант отдаёт 404 — проверено прямым запросом). Ответ содержит
  результат по каждому устройству группы отдельно; успехом считается только
  если `DONE` получили все.
- **`action_result.status`** приходит в верхнем регистре (`"DONE"`/`"ERROR"`),
  сравнение в `main.ts` регистронезависимое — иначе даже успешные действия
  считались бы ошибкой.
- **Ошибки IPC** — Electron не переносит произвольные поля Error через границу
  `ipcRenderer.invoke`/`contextBridge`, только `message`. Поэтому
  `shared/types.ts` кодирует структурированную ошибку (`kind`, `httpStatus`)
  прямо в `message` как JSON (`encodeError`/`decodeError`) — так UI надёжно
  отличает «нет токена» от «нет сети» от «ошибка устройства».
- **Поллинг** — при активной авторизации состояние дома обновляется каждые
  12 секунд по умолчанию (настраивается в ⚙️ «Настройки»); ручное обновление —
  кликом по индикатору статуса в шапке.
- **Системный трей** — приложение живёт в трее (`src/main/tray.ts`). Крестик
  на окне сворачивает в трей, а не закрывает процесс — полный выход только
  через пункт «Выход» в меню трея (ПКМ по иконке). В меню — список «последних
  действий»: последние 5 устройств/групп, у которых переключали
  включение/выключение (из самого приложения ИЛИ прямо из меню трея), с
  быстрым тумблером вкл/выкл без открытия окна. Список хранит и синхронизирует
  renderer (`hooks/useTrayMru.ts`) через `localStorage` + IPC
  (`tray:sync`) — сам main ничего не знает про устройства, только исполняет
  клик тем же HTTP-клиентом, что и обычные действия из окна.
- **Профиль в шапке** — вместо голой кнопки «Выйти» показывается логин
  подключённого аккаунта (`login.yandex.ru/info`, работает с любым валидным
  токеном без отдельного scope). Имени и аватара там нет — это требует
  отдельной группы прав «Яндекс ID» на oauth.yandex.ru, которую текущий
  сценарий (только `iot:view`/`iot:control`) не запрашивает; проверено прямым
  запросом с реальным токеном, Яндекс тихо не возвращает эти поля без scope.
  Клик по профилю открывает «Настройки»/«Выйти».
- **Стабильный порядок контролов** — API не гарантирует порядок capabilities
  ни между устройствами, ни между опросами одного устройства; без сортировки
  по фиксированному приоритету (`lib/capability.ts`) элементы карточек
  визуально «прыгали» местами при каждом обновлении.
- **Иконка приложения** — `build/icon.png` (1024×1024, автоматически
  подхватывается electron-builder'ом) нарисована по той же геометрии, что и
  32×32 иконка в трее, чтобы значок в панели задач и в трее выглядели
  одинаково.
- **Несколько домов** — `GET /user/info` отдаёт все дома, комнаты, устройства
  и группы аккаунта одним списком без разделения по дому; `household_id` есть
  у `Room`/`Device`/`DeviceGroup`, но не у `Scenario` (в реальном ответе API
  этого поля нет — сценарии показываются для всех домов сразу). Переключатель
  дома в сайдбаре — чисто клиентская фильтрация уже загруженных данных, без
  отдельного запроса к API.

## Дизайн UI

Интерфейс собран на компонентах HeroUI (https://heroui.com) поверх Tailwind
CSS, тёмная тема включена по умолчанию (`class="dark"` на `<html>`).
Конфигурация — `tailwind.config.js` / `postcss.config.js` в корне проекта.
